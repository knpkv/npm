# Control Center — Requirements

## Status and interpretation

Status: **Phase 2 approved; local cdx/cld provider requirement added by user during Phase 4 planning**

This document refines the approved feature instructions into testable requirements. `SHALL` and `MUST` are mandatory; `SHOULD` is mandatory unless Phase 3 documents a concrete reason and equivalent outcome. The committed prototype is a visual and interaction fixture, not a production data source or architectural template.

Two distinctions are normative:

- The acceptance fixture contains exactly six Jira items per release so relationship density is easy to judge, while the production domain and layout must support zero, one, six, or more items.
- `Active work` is a human-decision view containing issues, PRs, previews, runbooks, people, and agent activity. It must not collapse into a generic CodePipeline activity stream.

## FR1.x — Functional requirements

### Application shell and portfolio

- **FR1.1 — Canonical shell.** The web application SHALL expose Overview, Release, Active work, Items, Timeline, Settings, and stable full-entity routes. Navigation SHALL expose the active location, a context-aware agent entry point, plugin health/freshness, and a theme control without rendering competing primary pages.
- **FR1.2 — Startup and onboarding.** A fresh installation SHALL start in a recoverable onboarding state, configure at least one plugin connection through typed server APIs, verify its health, persist non-secret configuration, and reach a normalized portfolio without editing source files.
- **FR1.3 — Bird's-eye portfolio.** Overview SHALL show every normalized release with service, version, deterministic identity, readiness verdict, concise reason, Build/Verify/Production state, Jira/PR/pipeline relationship counts, evidence gaps, freshness, and named owner or approver.
- **FR1.4 — Portfolio filters.** Overview SHALL provide All, Need attention, Deploying, and Shipped filters whose labels, counts, rows, keyboard behavior, empty states, and URL state remain consistent as live data changes.
- **FR1.5 — Reference release states.** The reference portfolio SHALL contain six releases demonstrating blocked/cannot-ship, ready/can-ship, deploying, building, shipped, and held/needs-links states. Production state SHALL be domain data rather than a CSS tone or copied display verdict.

### Release and delivery views

- **FR1.6 — Preview before full view.** Activating any release row, verdict, or row action SHALL open a compact preview first. Only an explicit Open full view action may navigate to the canonical full release route.
- **FR1.7 — Release preview content.** Preview SHALL show verdict and reasons, freshness, deterministic identity, collaborators and roles, the release agent thread, Build/Verify/Production stages, the Jira work card, PR grouping, pipeline execution activity, Confluence evidence, gaps, and a state-specific primary action.
- **FR1.8 — Preview interaction.** Preview SHALL support pointer and keyboard invocation, initial focus, focus containment and restoration, Escape and explicit close, background inertness, scroll locking, and a reduced-motion transition fallback.
- **FR1.9 — Full release view.** Full view SHALL preserve preview context and add evidence-derived blockers, owners and approvers, complete stage state, actions, direct entity navigation, relationship evidence, missing links, runbook evidence, environments, and one durable release-scoped agent thread.
- **FR1.10 — Six-item relationship card.** The acceptance fixture SHALL show all six Jira items in one compact card. PRs SHALL be a distinct grouping dimension around those items, and pipeline executions/stages SHALL be a separate delivery dimension. Zero-link and many-to-many Jira↔PR relationships must remain explicit rather than hidden.
- **FR1.11 — Arbitrary release cardinality.** The production model and UI SHALL support zero, one, six, and more Jira items; zero or many PRs; multiple executions; and multiple target environments without truncation, false grouping, or layout failure.
- **FR1.12 — State-specific release actions.** Blocked releases SHALL offer investigate/repair; ready releases SHALL offer governed deployment confirmation; deploying releases SHALL offer watch; building releases SHALL offer notification; shipped releases SHALL offer acknowledgement; held releases SHALL offer trace repair. Completion SHALL update every dependent view from one server-authoritative result and SHALL be idempotent.
- **FR1.13 — Active work.** Active work SHALL center the current human decision, preview/check evidence, next action, collaborators, six-item work card, Jira↔PR grouping, review lifecycle, agent note, recent cross-service activity, and current runbook. The reference lifecycle SHALL cover review not requested → requested → reviewed/ready to merge.
- **FR1.14 — Items.** Items SHALL search and filter all normalized delivery objects by text, service/type, status, and owner; show result counts, source, owner, freshness, status, recoverable empty/error states, and full-view navigation; and support starting a delivery trace from a Jira issue, PR, or release.
- **FR1.15 — Timeline.** Timeline SHALL merge attributable human, agent, plugin, and system events in stable order; filter by actor and time range; deep-link to related objects/findings; and export CSV and JSON with optional raw identifiers and agent-action detail.

### Relationships, evidence, and actions

- **FR1.16 — Relationship inspection.** A user SHALL inspect every typed relationship with source, target, direction, lifecycle, provenance, confidence or rationale, freshness, and supporting evidence.
- **FR1.17 — Relationship repair.** Repair SHALL show the current gap, candidate targets, source and confidence, exact proposed relationship, impacted release/entities, evidence revision, and required permission before confirmation. Staging is not mutation; applying one repair SHALL not mutate unrelated relationships or releases.
- **FR1.18 — Readiness recomputation.** A relationship, approval, check, execution, deployment, evidence-freshness, or plugin-health change SHALL recompute only affected readiness assessments and retain the previous assessment for audit.
- **FR1.19 — Governed remote mutations.** Before any Jira, Confluence, CodeCommit, CodePipeline, or Clockify write, the UI SHALL show the action, exact target and expected revision, impact, proposed payload/change, evidence, permission/policy, and confirmation. Denial, expiry, cancellation, stale evidence, or changed payload SHALL produce no vendor mutation.
- **FR1.20 — Action outcomes.** Every proposed, authorized, started, succeeded, failed, denied, expired, or cancelled governed action SHALL produce an attributable audit event and an idempotent, persisted outcome visible in the initiating and affected views.

### Full service views

- **FR1.21 — Shared entity shell.** Jira, CodeCommit PR, Confluence, CodePipeline, and Clockify routes SHALL use the production design system and show service identity, large verdict, title/status, freshness, people with explicit roles, a navigable cross-service relationship chain, native content, facts, activity, contextual agent, and governed actions.
- **FR1.22 — Jira issue.** Jira SHALL include structured fields, assignee/owner, priority, estimate, release/workstream, rich description read/edit/cancel/save, independently togglable acceptance criteria, comments, threaded replies, validated composer, human and synchronized history, delivery evidence, and governed transition/link/approval actions. Successful edits SHALL persist and append history/audit.
- **FR1.23 — CodeCommit PR.** A PR SHALL include author, source/base branch, immutable head commit, diff summary, commits, checks, reviewers, approvers, human review state, linked Jira/pipeline/release evidence, request-review/request-changes/approval state, and governed merge-capable actions when supported. Its Files view SHALL load the complete PR diff across every changed file and render it with `@pierre/diffs/react`, including file tree and changed-file status, split and stacked layouts, syntax highlighting, line numbers, inline change highlighting, wrapped/unwrapped lines, unchanged-context expansion, line selection, and annotation/finding anchors. Added, modified, deleted, renamed, binary, generated, too-large, and unavailable files SHALL have explicit states; large PRs SHALL remain complete through lazy loading and virtualization rather than silent truncation.
- **FR1.24 — Sandboxed agent review.** Starting agent review SHALL resolve a trusted PR head server-side, create an isolated sandbox pinned to that commit, progress through checkout → analysis → findings, continue after navigation or UI close, stream persisted progress, return file-specific findings, and let a human record approve or request changes. Agent recommendation SHALL never impersonate human approval.
- **FR1.25 — CodePipeline execution.** CodePipeline SHALL include pipeline/execution identity, trigger and revision, artifact and target, duration, stage states, logs, verified/downloadable artifact references, environment and rollout state, operators and deployment approver, related PR/release/runbook, and governed retry/start-deploy/watch actions. Retry SHALL create a new linked execution and preserve the failed one.
- **FR1.26 — Confluence page.** Confluence SHALL include readable current content, scope/prerequisites/commands, safe copy interaction, revision and activity history, owner/contributors/approver/watchers, related Jira/pipeline/release evidence, freshness, and governed update/publish. Superseded revisions SHALL remain auditable.
- **FR1.27 — Clockify view.** Clockify SHALL include entries and release/ticket rollups, duration/date/project/billable state, contributors, ticket/PR/release associations, approval/attribution and evidence state, and attributable activity. Empty, inferred, unattributed, pending, approved, stale, and error states SHALL be representable.

### People, agents, plugins, and presentation

- **FR1.28 — Human collaboration.** Relevant portfolio, release, Active work, and entity views SHALL show real people with avatar or accessible fallback, name, and explicit role: release/change/issue/page owner, author, contributor, reviewer, operator, or deployment/merge/workspace approver. Roles and lifecycle SHALL be persisted data, not inferred from position.
- **FR1.29 — Contextual agent.** Every primary page SHALL expose an agent whose launcher and thread state the exact portfolio, release, Active work, or entity context and enumerate the evidence included in a request.
- **FR1.30 — Release-scoped threads.** Every release SHALL have one durable, isolated thread for human, agent, and system messages and actions including evidence refresh, description update, checks, blocker summary, and PR review. Background runs SHALL support progress, cancellation, terminal state, restart reconciliation, and strict release isolation.
- **FR1.31 — Agent provider boundary.** The application SHALL define a typed provider/job contract for streaming output, tool/action proposals, cancellation, usage metadata when available, and structured results; SHALL include a deterministic fake provider for tests; and SHALL support configured providers without coupling domain code to them. Local Codex and Claude CLI integrations SHALL live in separate publishable `@knpkv/cdx` and `@knpkv/cld` packages. Each SHALL expose an Effect AI `LanguageModel`-compatible layer with typed executable/version discovery, text and structured generation, streaming/progress, cancellation, capability reporting, bounded process I/O, and Schema-decoded output. Unsupported Effect AI capabilities SHALL fail explicitly rather than being simulated or silently ignored.
- **FR1.32 — Plugin administration.** Settings SHALL list all five sources with distinct identity, endpoint/workspace metadata, capability/version, connected/syncing/degraded/failed/disabled state, freshness, safe diagnostics, health refresh, configuration, enable/disable, and permission management.
- **FR1.33 — Workspace settings.** Relationship inference, sync cadence, evidence retention, failure investigation, Jira comments, pipeline retry, agent policy, and presentation preferences SHALL be schema-validated, versioned, concurrency-safe, persisted on the server where shared, and provide dirty/saving/saved/conflict/error feedback.
- **FR1.34 — Search, share, and export.** Global search SHALL find releases and service entities with a no-results state. Share SHALL create a stable authorized link. Export SHALL preserve source attribution and evidence identity without exposing credentials.
- **FR1.35 — Stable routing and origin.** Canonical path-based URLs SHALL survive direct load and refresh. Back navigation SHALL retain the exact portfolio/release preview/full/Active work/Items origin, filters, and reasonable scroll state. Unknown or removed resources SHALL show a recoverable not-found state and SHALL never substitute a demo entity.
- **FR1.36 — Release identity.** Each immutable release ID SHALL deterministically produce a stable short codename and exactly three distinct SVG symbol tiles. Different versions of one service SHALL be distinguishable; the algorithm SHALL use canonical release identity and expose accessible text and an algorithm version.
- **FR1.37 — Design system catalog.** Production pages SHALL be composed from documented and independently tested rly tokens and components for typography, themes, spacing, surfaces, buttons, service badges, status/verdicts, avatars/collaborators, tabs, relationship chains, sigils, dialogs/sheets, agent messages/threads, timelines, tables/lists, forms, loading/empty/error states, and diff primitives. rly SHALL publish a static Storybook catalog linked from its documentation page, with every public component represented by interactive stories covering its declared variants, meaningful states, light/dark/forced-colors, responsive widths, reduced motion, density, keyboard behavior, and accessibility checks. rly SHALL also publish a generated, machine-readable component/variant/accessibility registry for agents and tooling without providing a runtime JSON-to-UI execution path.
- **FR1.38 — Live server state.** Portfolio, entities, relationships, evidence, settings, approvals, reviews, jobs, threads, and audit SHALL come from typed APIs and durable server state. Plugin sync, pipelines, and agents SHALL update without full reload and reconcile after reconnect.

## NFR2.x — Non-functional requirements

- **NFR2.1 — Accessibility.** Primary pages and flows SHALL conform to WCAG 2.2 AA. Automated checks SHALL report no serious or critical violations, and keyboard plus screen-reader acceptance SHALL be tested because automation alone is insufficient.
- **NFR2.2 — Keyboard and focus.** Every action SHALL be keyboard reachable. Dialogs and sheets SHALL set, contain, and restore focus; tabs SHALL support arrow navigation; live updates SHALL not steal focus.
- **NFR2.3 — Non-visual equivalence.** Readiness, relationship state, plugin health, approvals, and diff additions/removals SHALL not rely only on color, icon, avatar, or spatial placement. The graph SHALL have an equivalent semantic list/table view.
- **NFR2.4 — Reflow, themes, and motion.** Every flow SHALL remain usable at 200% zoom and 320 CSS px, in light, dark, and forced-colors modes. Reduced motion SHALL remove nonessential transitions without hiding state change.
- **NFR2.5 — Interaction performance.** With the documented reference fixture, cached portfolio data SHOULD be usable within two seconds of navigation on the Phase 3 benchmark machine, local interaction feedback SHOULD appear within 100 ms, and long work SHALL become a persisted background job immediately.
- **NFR2.6 — Bounded resources without data loss.** Pagination, queries, live-event payloads, logs, diff requests, comments, exports, agent output, and sandbox storage SHALL have explicit server bounds. Bounds SHALL use pagination, ranges, streaming, virtualization, or explicit unavailable states and SHALL NOT silently truncate the complete PR changed-file set required by FR1.23. Event storms SHALL be coalesced or backpressured, and slow clients SHALL not block other work.
- **NFR2.7 — Partial-failure availability.** Timeout, authentication failure, rate limiting, malformed data, or outage in one plugin SHALL not fail unrelated routes, erase cached normalized data, or disable unrelated actions. Affected data SHALL expose staleness and actionable health.
- **NFR2.8 — Live-update resilience.** Clients SHALL reconnect with capped exponential backoff and jitter, resume by event cursor or fetch an authoritative snapshot, ignore duplicate/out-of-order events, and expose connected, reconnecting, offline, and stale states.
- **NFR2.9 — Restart resilience.** Graceful shutdown SHALL stop new mutations, flush durable writes, and interrupt owned work. Startup SHALL deterministically reconcile syncs, agent jobs, permission prompts, and sandboxes so no job remains indefinitely running.
- **NFR2.10 — Safe binding.** Default binding SHALL be loopback. Non-loopback/LAN binding SHALL require explicit configuration, authenticate application sessions, enforce configured origins/hosts, and print the effective reachable host and port without silently selecting a misleading port.
- **NFR2.11 — Web security.** The server SHALL enforce an origin/host allowlist, mutation CSRF protection suitable for the session design, body/request limits, static-path containment, CSP, frame-ancestor protection, MIME-sniffing protection, and allowlist sanitization of vendor/Markdown/diff/agent content and URLs.
- **NFR2.12 — Credentials and privacy.** AWS and vendor credentials, tokens, session secrets, sandbox environment values, and secret-bearing errors SHALL never enter browser responses/storage, live events, normalized entity tables, agent prompts, audit context, or unredacted logs. The UI may expose only non-secret configuration metadata.
- **NFR2.13 — Governance default.** Missing policy SHALL fail closed for remote writes. Authorization SHALL be scoped to actor/session, workspace, plugin/action, exact targets, payload digest, evidence revision, and expiry.
- **NFR2.14 — Audit integrity.** Governed action events SHALL be append-only. Audit export, deletion, retention, and policy changes SHALL themselves be governed and audited. Remote mutation SHALL fail if its audit intent cannot be durably recorded.
- **NFR2.15 — Review integrity.** Human approvals and agent recommendations SHALL use distinct actor types. Approval SHALL bind to entity revision/commit/environment and become stale or invalid when that target changes.
- **NFR2.16 — Sandbox isolation.** PR review SHALL run non-root in a pinned/allowlisted image with no privileged mode, dropped capabilities, resource/time/disk/log limits, isolated workspace, default-deny network, and no host credentials or arbitrary mounts. Exceptions SHALL be separately governed.
- **NFR2.17 — Sandbox access.** Checkout target SHALL be derived from a trusted PR record and immutable commit. Any sandbox web port SHALL bind loopback and require a short-lived scoped token; allocation SHALL be collision-safe; findings and paths SHALL be constrained to the sandbox workspace.
- **NFR2.18 — Durability.** Durable mutations SHALL be transactional, idempotent, and crash-safe. Ordered migrations and backup/restore SHALL be tested from the previous released schema. Invalid records SHALL be quarantined without resetting unrelated state.
- **NFR2.19 — Maintainability.** Vendor shapes SHALL not leak into core or UI contracts, exported APIs SHALL be documented, page-specific copies of design primitives SHALL be rejected, and architecture SHALL preserve browser/server dependency separation. rly SHALL be an independent browser-safe package and SHALL NOT depend on Control Center domain, API, server, or vendor packages. Semantic token usage, package boundaries, public variants, generated registry completeness, and source/export/build consistency SHALL be mechanically enforced.
- **NFR2.20 — Browser-process discipline.** Component tests SHALL prefer a DOM test environment; browser E2E SHALL run through an explicit command with one bounded browser worker by default and deterministic process/context teardown.
- **NFR2.21 — Local agent process safety.** Local agent wrappers SHALL spawn only a configured/discovered allowlisted executable with argument arrays, validated working directory, bounded prompt/stdout/stderr/time, minimal allowlisted environment, no interactive TTY, scoped interruption and forced cleanup. They SHALL NOT enable permission/sandbox bypass flags or inherit hooks, plugins, MCP servers, browser control, arbitrary settings, or additional writable directories by default. Any less-isolated local-profile mode SHALL be an explicit governed capability with visible diagnostics and SHALL remain unavailable to unattended vendor mutations.

## TC3.x — Technical constraints

- **TC3.1 — Packages.** The application SHALL live at `packages/control-center` and publish as ESM package `@knpkv/control-center`. The reusable browser-safe **rly** design system SHALL live at `packages/rly` and publish as ESM package `@knpkv/rly`. The local Effect AI wrappers SHALL live at `packages/cdx` / `@knpkv/cdx` and `packages/cld` / `@knpkv/cld`. All four SHALL define `main`, `types`, `exports`, `publishConfig`, `files`, and explicit applicable `dev`, `build`, `check`, `lint`, and `test` scripts; the application additionally owns `start` and `test:e2e`.
- **TC3.2 — TypeScript.** The package SHALL extend repository TypeScript configuration, use strict NodeNext/ES2022 and React JSX settings, exact optional properties, unchecked-index protection, unused checks, `.js` relative import specifiers, and separate browser/server dependency boundaries.
- **TC3.3 — Type safety.** Production TypeScript SHALL not use `any`, assertions, non-null assertions, enums, default exports in shared modules, `@ts-ignore`, `eval`, or `new Function`. Public functions SHALL have explicit return types and exported APIs SHALL have JSDoc.
- **TC3.4 — Effect version and idioms.** `effect` and `@effect/*` SHALL remain aligned with the repository beta version. Before implementation, `repos/effect/LLMS.md` SHALL be read and relevant APIs verified in `repos/effect/packages`. Services SHALL use `Context.Service`, explicit Layers, tagged errors, and typed error channels.
- **TC3.5 — Effect Platform boundaries.** Effect code SHALL use Effect Platform HTTP, configuration, filesystem, clock, crypto, process, and scheduling services. Raw host APIs are allowed only at documented React/Vite/framework boundaries.
- **TC3.6 — Boundary decoding.** HTTP payloads, environment/configuration, persistence, plugin data/events, live events, agent output, and sandbox/process results SHALL be decoded from `unknown` with Schema before entering trusted domain state.
- **TC3.7 — Vendor isolation.** Control Center core SHALL depend only on vendor-neutral plugin contracts and domain types. Authentication, pagination, retries, rate limits, errors, and vendor response shapes SHALL remain in adapters.
- **TC3.8 — External-call policy.** Vendor and agent calls SHALL support cancellation and configurable timeouts. Retries SHALL be bounded to safe/idempotent operations, honor `Retry-After`, add jitter, and surface terminal authentication/rate-limit health.
- **TC3.9 — Persistence.** Production domain state SHALL use a migration-managed transactional store separate from the CodeCommit-specific cache. Prototype collections and browser `localStorage` SHALL not be production state; browser storage is limited to non-sensitive user presentation preferences.
- **TC3.10 — Typed API and streaming.** Server contracts SHALL use Effect Schema and Effect HTTP APIs in a browser-safe module. Same-origin authenticated SSE SHOULD carry versioned, bounded events with initial snapshot, cursor, heartbeat, reconnect/resume, and scoped authorization unless Phase 3 documents an equivalent transport.
- **TC3.11 — Runtime.** The production server SHALL support the repository CI runtime (Node 24 with `@effect/platform-node`) unless Phase 3 explicitly adds and validates another runtime. Host, port, allowed origins, session/authentication, data path, and credential references SHALL be decoded configuration.
- **TC3.12 — Content safety.** Jira/Confluence rich content, Markdown, diffs, logs, and agent output SHALL pass through an allowlisted representation and sanitization policy; untrusted HTML, scriptable SVG, unsafe URLs, and external credential-bearing media SHALL not bypass it.
- **TC3.13 — Supervised jobs.** Plugin sync, agent work, sandbox work, and long actions SHALL be supervised/scoped jobs with durable ID, ownership, progress, cancellation, timeout, retry policy, and terminal state rather than detached fibers.
- **TC3.14 — Diff rendering.** Production PR diffs SHALL use the supported React entry point from `@pierre/diffs/react`. Large multi-file diffs SHALL use its virtualization facilities and SHOULD offload highlighting through its worker-pool API; the implementation SHALL not fork or recreate the library's renderer as page-local markup.
- **TC3.15 — Test discovery and package integrity.** All four packages SHALL include root-discoverable Vitest configuration and applicable lint coverage for `.ts`/`.tsx`; rly SHALL include component DOM setup, post-build export/declaration/CSS validation, strict packed-consumer type/import tests, packed-registry resolution/schema validation, and bounded visual tests; cdx/cld SHALL include packed-consumer and fake-executable protocol/process contract tests; the application SHALL include explicit bounded browser E2E. Root tooling SHALL be updated so UI and process-provider code cannot escape CI checks.
- **TC3.16 — Release validation.** Validation SHALL include unit, component, contract, server integration, migration, failure-injection, accessibility, and E2E tests plus repository format, ESLint, AST, type-check, build, changeset, dependency audit, and frozen-lockfile installation. Every confirmed agent-review finding SHALL include an implementation-ready prevention proposal classified as ast-grep, ESLint, type-check, test, instruction, or justified none. The proposal SHALL name an existing rule or configuration to extend before inventing another, target files and covered paths, the matcher or invariant, one rejected failure fixture, one allowed nearby fixture, exclusions, and false-positive risks. Stable guardrails SHALL ship with the remediation; a brittle proposal SHALL be replaced with the next durable layer or retain a recorded rationale.

## DR4.x — Data requirements

- **DR4.1 — Workspace scope.** Every durable entity, relationship, evidence item, job, agent message, permission decision, audit event, and live event SHALL carry immutable workspace scope; release/entity scope SHALL be explicit and enforced in repositories.
- **DR4.2 — Canonical identity.** Vendor objects SHALL retain plugin/connection ID, vendor immutable ID, canonical Control Center ID, source revision/version and URL, first/last observed times, and normalization schema version. Display names SHALL never be join keys.
- **DR4.3 — Release aggregate.** A release SHALL contain canonical ID, service/version display data, targets/environments, lifecycle and derived readiness, reasons, stage summary, freshness, role assignments, sigil algorithm/version, relationship/evidence references, blockers/gaps, and timestamps.
- **DR4.4 — Relationship record.** A relationship SHALL contain ID, typed source and target, kind/direction, lifecycle (`missing`, `inferred`, `proposed`, `verified`, `governed`, `rejected`, `superseded`), provenance, confidence/rationale, evidence, actor, release/environment scope, timestamps, version, and supersession/revocation history.
- **DR4.5 — Immutable evidence.** Evidence SHALL be append-only and content- or revision-addressed with source object/revision, observation and ingest times, normalized claims, integrity/version, retention metadata, freshness, and verifier. Corrections SHALL supersede rather than overwrite.
- **DR4.6 — Readiness assessment.** Readiness SHALL store verdict, rule/derivation version, evaluated time, environment, required and verified facts, blockers/warnings/gaps, source freshness, and exact evidence IDs. Prior assessments SHALL be retained.
- **DR4.7 — Entity read models.** A shared read model SHALL cover ID, kind, service, title, status/verdict, facts, relationships, activity, actions, freshness, and people while native extensions preserve all data required by FR1.22–FR1.27.
- **DR4.8 — People and roles.** Person SHALL contain canonical ID, display name, avatar reference/fallback, source identities, and active state. Role assignment SHALL bind a person or agent to exact workspace/release/entity/environment, role, lifecycle, and time.
- **DR4.9 — Reviews and approvals.** Each reviewer/approver decision SHALL store actor type and role, target revision/environment, decision, rationale, timestamp, evidence, expiry, and invalidation. Multiple decisions SHALL remain separate rather than collapse to booleans.
- **DR4.10 — Agent threads and jobs.** A thread SHALL be keyed by `(workspaceId, releaseId)`. Messages and jobs SHALL record immutable identity/order, actor, exact context and evidence snapshot, action/provider/tool metadata, sandbox/revision when relevant, progress, result/findings, permission, error/cancellation, and correlation ID.
- **DR4.11 — Governed action.** An action SHALL store ID, idempotency key, canonical payload digest, plugin/capability version, actor/session, exact target and expected revision, proposal/impact, evidence, authorization, execution attempts, timestamps, and outcome. Duplicate idempotency keys SHALL not execute twice.
- **DR4.12 — Live-event envelope.** Live events SHALL include schema version, event ID/cursor, workspace, optional plugin/release/entity/job IDs, type, UTC occurrence and ingest times, causation/correlation IDs, and decoded payload.
- **DR4.13 — Plugin health and cache.** Plugin state SHALL record type and contract version, enabled/configured state, capabilities, health/failure class, last attempt/success, next retry, rate-limit data, sync checkpoint, cache freshness, and safe diagnostics without credentials.
- **DR4.14 — Credentials.** Plugin configuration SHALL store credential references only. Secret values SHALL live in separate owner-protected storage, use redacted in-memory representation, and be structurally impossible to serialize through normal API/domain schemas.
- **DR4.15 — Settings and concurrency.** Shared settings and mutable records SHALL carry revision/ETag or equivalent concurrency token. Stale writes SHALL return conflict data rather than silently overwrite descriptions, comments, relationships, approvals, or settings.
- **DR4.16 — Retention and privacy.** Retention SHALL be documented and independently configurable for audit, agent content, sandbox logs/workspaces, plugin cache, normalized content, and evidence. Cleanup SHALL be bounded and audited. Person, comment, time, avatar, and prompt data SHALL be minimized and attributable.
- **DR4.17 — Sandbox state.** Sandbox state SHALL include trusted checkout commit, pinned image digest, policy profile, resource limits, owner/session, loopback port/token reference if any, progress/status, activity/expiry, bounded redacted logs, findings, and cleanup outcome.
- **DR4.18 — Invalid data.** Malformed plugin or persisted records SHALL be quarantined as bounded redacted diagnostics; dangling/deleted targets SHALL be explicit; invalid data SHALL not enter trusted collections, crash startup, or replace the last valid cache.
- **DR4.19 — Release identity projection.** Codename and exactly three distinct symbols SHALL derive from canonical release identity plus algorithm version and SHALL remain stable across clients, restarts, and migrations.
- **DR4.20 — Freshness.** Every normalized object SHALL expose source-observed time, synchronized time, stale threshold, cache provenance, and plugin health so UI can distinguish current, cached/stale, missing, and unavailable.

## IR5.x — Integration requirements

- **IR5.1 — Plugin contract.** A versioned vendor-neutral contract SHALL cover discovery, configuration metadata, capabilities, health, normalized reads, checkpoints/events, and governed action proposals/execution. Plugins SHALL be independently enabled, disabled, and failed.
- **IR5.2 — CodeCommit.** The CodeCommit adapter SHALL reuse public `@knpkv/codecommit-core` account/configuration, PR, cache/event, permission, and sandbox capabilities where suitable. The owning package SHALL expose typed, paginated APIs for the complete changed-file inventory, before/after content or unified patch data, file status and paths, revisions, binary/generated/oversize metadata, and stable line anchors required by `@pierre/diffs`; it SHALL preserve CodeCommit pagination and explicit provider limits rather than returning aggregate diff counts only. Missing comment/write, approval-state, merge, and agent-review primitives SHALL likewise be added through supported public exports rather than deep imports.
- **IR5.3 — Jira.** The Jira adapter SHALL reuse `@knpkv/jira-api-client` and reusable higher-level issue/version services. It SHALL add or expose supported APIs for changelog/history, custom/acceptance fields, avatars/roles, description edits, comments/replies, transitions, versions, contributors, and related-work links.
- **IR5.4 — Confluence.** The Confluence adapter SHALL reuse `@knpkv/confluence-api-client` and `@knpkv/confluence-to-markdown` public page, version, user, attachment, and conversion capabilities. Release evidence and approval semantics SHALL remain Control Center domain data.
- **IR5.5 — Clockify.** The Clockify adapter SHALL reuse `@knpkv/clockify-api-client` and promote or re-home reusable credential, Jira-key association, and rollup logic currently internal to `@knpkv/jira-clockify`. Approval/evidence state SHALL remain Control Center data.
- **IR5.6 — CodePipeline.** A new adapter SHALL directly depend on and Schema-wrap supported CodePipeline operations, credentials, retries, pagination, and errors. It SHALL cover reads and governed start/stop/manual-approval actions; CodeBuild, CloudWatch Logs, or S3 access needed for logs/artifacts SHALL remain behind the same adapter boundary.
- **IR5.7 — Authentication and profiles.** Server configuration SHALL reuse public Atlassian profile/token storage and AWS profile/region support where applicable. Browser APIs SHALL exchange credential/profile references and redacted status, never secrets.
- **IR5.8 — Governance.** Existing CodeCommit permission-prompt and audit patterns MAY be reused, but production governance SHALL use a vendor-neutral, authenticated-session-scoped action envelope and append-only proposal-to-outcome audit.
- **IR5.9 — Agents.** Because no agent runtime currently exists, Control Center SHALL introduce a provider/job abstraction, deterministic test provider, release-scoped context builder, supervised job lifecycle, sandbox review integration, structured findings, cancellation, and governed action handoff. `@knpkv/cdx` and `@knpkv/cld` SHALL independently adapt supported local Codex and Claude CLI structured modes into Effect AI `LanguageModel` layers; Control Center SHALL consume them through supported public exports and a provider registry alongside the HTTP provider.
- **IR5.10 — Persistence.** The package SHALL own a normalized delivery-graph database and migrations. CodeCommit's existing cache MAY be read through its public service but SHALL not become the generic graph or Control Center audit store.
- **IR5.11 — Typed HTTP.** Client-safe Effect HTTP API contracts and handlers SHALL cover portfolio, releases, entities, relationships/evidence, plugins/health, settings, agent threads/jobs, approvals/reviews, governed actions, audit, and live updates.
- **IR5.12 — Partial failure.** Adapter timeout, 401/403, 429, malformed payload, or outage SHALL map to scoped typed health/error state, preserve prior valid normalized data, and not terminate unrelated plugin layers or API groups.
- **IR5.13 — Prototype boundary.** Production runtime SHALL not import prototype mock collections, components, cascade CSS, or `localStorage` state. Prototype fixtures MAY inform test fixtures, screenshots, and visual acceptance.
- **IR5.14 — Documentation.** `packages/docs` SHALL publish indexed, searchable documentation-site pages for `@knpkv/control-center`, `@knpkv/rly`, `@knpkv/cdx`, and `@knpkv/cld`, not only package READMEs. Pages SHALL cover installation, supported exports, Effect AI/rly usage examples, configuration, safe capability/profile behavior, the explicit cdx local smoke command, secure LAN startup, plugin setup, architecture, design catalog, agent/governance behavior, recovery, and troubleshooting. Package indexes/navigation SHALL link every page, and code/command examples SHALL be checked where practical.
- **IR5.15 — Existing-package changes.** Any public capability added to CodeCommit, Jira, Confluence, Clockify, Atlassian common, or docs SHALL be placed in its owning package, exported through supported entry points, tested there, and accompanied by a changeset when publishable behavior changes.

## SC7.x — Success criteria and acceptance evidence

- **SC7.1 — Fresh installation.** An integration/E2E test configures one plugin, starts the server, loads the UI, shows plugin health, and renders normalized release data from durable APIs.
- **SC7.2 — Portfolio.** With the reference fixture, Overview shows six releases in six distinct states with correct filters/counts, people, sigils, relationship/gap counts, stages, and freshness.
- **SC7.3 — Preview contract.** Pointer or keyboard activation of every release opens preview first with the required six-item relationship content and focus behavior; only Open full view navigates to the canonical route.
- **SC7.4 — Routing and back.** Direct load/refresh of release and entity URLs restores the same data. Release → Jira → PR and Items/Active work origins return through exact prior contexts without substituting a default entity.
- **SC7.5 — Six-item clarity.** The payments fixture visibly shows all six Jira items, two PR groupings covering five, one unlinked item, pipeline execution as a separate dimension, runbook evidence, collaborators, and environment state. Every object opens full view.
- **SC7.6 — Relationship repair.** Applying one of two missing relationships reduces only that gap and audits it; applying the second recomputes readiness. Stale evidence or repeated idempotency key causes no duplicate mutation.
- **SC7.7 — Pipeline retry.** Retrying failed execution `#1842` creates and links a new execution while preserving `#1842`; repeated submission executes once and produces a complete audit chain.
- **SC7.8 — Active work.** OPS-428 centers human review rather than generic pipeline streaming, shows six Jira items, two PRs, preview evidence, people, runbook, and agent activity; request and completion transitions persist through refresh/restart.
- **SC7.9 — Jira lifecycle.** Description save/cancel, criteria, comment/reply validation, threaded display, and history work and persist. A concurrent stale edit returns a recoverable conflict rather than silently overwrite.
- **SC7.10 — Complete PR diff and agent review.** A fixture containing added, modified, deleted, renamed, binary, generated, and large files proves the UI exposes every changed file without silent truncation and renders supported text through `@pierre/diffs/react` in split and stacked modes with syntax, context expansion, line selection, and stable finding anchors. Agent review checks out the trusted immutable PR head in an isolated sandbox, survives UI close, streams progress, attaches file/line findings to those anchors, supports human decision, persists evidence, and cleans up safely.
- **SC7.11 — Agent isolation.** Two release threads and browser sessions receive only their authorized messages, evidence, prompts, and events. Foreign release IDs are rejected and restart restores correct job terminal state.
- **SC7.12 — Governed mutation.** Changed payload/revision after authorization is rejected; deny/timeout/cancel causes zero vendor calls; same idempotency key executes once; success and failure append the full audit lifecycle.
- **SC7.13 — Full service routes.** Canonical Jira, PR, Confluence, CodePipeline, and Clockify URLs load from typed APIs, show common collaboration/relationship/agent shell plus required native data, retain state on refresh, and isolate loading/empty/stale/error/not-found states.
- **SC7.14 — Plugin partial failure.** For every adapter, timeout, 401/403, 429 with `Retry-After`, malformed payload, and outage leave unrelated routes usable, retain and mark cached data stale, block unsafe actions, and reconcile without duplicate events after recovery.
- **SC7.15 — Live recovery.** Disconnecting during multiple pipeline/agent/sync events and reconnecting produces the same final state as an authoritative snapshot, without duplicate action or readiness regression, and exposes connection state accessibly.
- **SC7.16 — Restart and corruption.** Kill/restart during sync, review, permission wait, and sandbox creation leaves each job resumed, reconciled, failed, or cancelled. Migration and transaction fault tests preserve valid state while quarantining a malformed record.
- **SC7.17 — LAN security.** Default server accepts loopback only. Explicit non-loopback binding exposes the configured port to a second machine, reports usable URLs, authenticates UI/API/live updates, rejects disallowed Host/Origin/CSRF, and leaks no credential.
- **SC7.18 — Sandbox containment.** Security tests prove non-root execution, quotas, pinned image/commit, no host credentials or unapproved paths/network, no unauthenticated LAN port, collision-safe access, and cleanup after timeout/restart.
- **SC7.19 — Content and secret safety.** Hostile Jira/Confluence/Markdown/SVG/diff/agent content cannot execute. Seeded AWS/vendor/session secrets do not appear in HTML, API, live events, non-secret DB tables, audit export, browser state, application logs, or sandbox logs.
- **SC7.20 — Accessibility and presentation.** Keyboard-only E2E covers portfolio → preview → full → entity, governed actions, Jira editing, PR review, settings, and agent flows. Light/dark/forced colors, reduced motion, 200% zoom, and 320 px reflow preserve every action; automated checks have no serious/critical findings and a manual screen-reader checklist passes.
- **SC7.21 — Performance and bounds.** The documented benchmark meets NFR2.5, and large-fixture tests prove bounded pagination/log/diff/event/agent data plus stable behavior during an event burst and slow client.
- **SC7.22 — Design-system reuse and Storybook.** Component tests, generated-registry validation, packed-consumer tests, package-boundary tests, semantic-token lint, and source review prove all production pages consume documented `@knpkv/rly` exports, rly has no application/vendor dependency, public build artifacts match its manifest, and pages neither import prototype runtime nor duplicate covered primitives. The static Storybook build SHALL succeed with a one-to-one manifest-to-story check, navigable autogenerated documentation, interaction/accessibility tests, and the bounded high-risk light/dark desktop/mobile visual matrix; the rly documentation page SHALL link the published catalog.
- **SC7.23 — Complete validation.** Focused package and changed-package checks pass, followed by frozen install, `pnpm format`, `pnpm lint`, `pnpm lint:ast`, `pnpm build`, `pnpm check`, `pnpm test --run`, E2E, changeset status, documentation checks, and dependency audit. The committed review record links every accepted finding to its shipped static-analysis, type-check, test, or instruction guardrail, or to a concrete false-positive rationale for justified none.
- **SC7.24 — Local Effect AI wrappers.** Fake-executable contract suites prove `@knpkv/cdx` and `@knpkv/cld` discover executable/version/capabilities, expose `LanguageModel.generateText`, structured generation, and `streamText`, decode JSONL/stream-JSON incrementally, propagate cancellation, bound I/O, redact environment/authentication failures, and leave no child process. Each package SHALL preserve the complete `LanguageModel.Service` interface while guarding untouched public call options before `LanguageModel.make` can normalize or consume them, with an additional provider-hook guard over normalized options. Outer-guard matrix tests SHALL cover the supported single user/text prompt with text and unnamed JSON response formats, then prove system/multiple/history/assistant/tool messages; file/image/reasoning/tool/approval parts; explicit structured-output `objectName`; non-empty provider options; any supplied toolkit including empty and effectful toolkits; explicit `toolChoice: "auto"` without a toolkit and every other non-`none` tool choice; supplied `concurrency`; supplied `disableToolCallResolution`; and approval history paired with an effectful toolkit fail with typed `AiError` before delegation rather than being flattened, simulated, normalized away, or ignored. Rejection tests SHALL assert zero CLI invocations and zero toolkit-construction/tool-handler effects. A public-service integration test SHALL seed Effect AI's `ResponseIdTracker` and submit the history/assistant shape required for continuation derivation, proving the outer guard rejects that unsupported history before delegation with zero effects. Separately, a direct inner-provider-hook contract test SHALL supply normalized `previousResponseId`/`incrementalPrompt` options and prove a non-fallback typed `AiError` occurs before any CLI invocation. In addition, a deliberate `@knpkv/cdx` local smoke command SHALL exercise the installed authenticated Codex CLI through the packaged Effect AI layer in an ephemeral read-only temporary repository, validate structured and streaming output, record the CLI version, enforce time/output bounds, and prove cleanup. It SHALL be required acceptance evidence on the development machine but SHALL NOT run automatically in normal CI, pre-commit, or unattended environments. The equivalent real Claude smoke remains optional unless separately enabled.
- **SC7.25 — New-package documentation site.** The Astro/Starlight documentation build SHALL expose navigable/searchable Control Center, rly, cdx, and cld pages plus package-index entries. Each page SHALL match its packed public exports and current commands; cdx/cld examples SHALL show Effect AI layers, safety defaults, capability errors, cancellation, and local smoke behavior without suggesting dangerous bypass flags or embedding credentials.
