---
title: Conventions
description: Shared command conventions across product binaries.
---

These conventions keep command usage predictable for humans and agents. They are source-backed by the CLI implementations under each package's `src/commands` or `src/cli` directory.

## Namespaces

Commands are organized by resource:

| Namespace                                                      | Meaning                                                |
| -------------------------------------------------------------- | ------------------------------------------------------ |
| `auth`                                                         | Authentication setup, login, logout, and status checks |
| `config`                                                       | Local tool configuration                               |
| `issue`, `version`, `page`, `workspace`, `sync`, `pr`, `timer` | Domain resources                                       |
| `skills install`                                               | Bundled agent skill installation                       |

Legacy top-level aliases are intentionally removed. Use `jira issue get`, not `jira get`; use `jcf timer start`, not `jcf start`; use `confluence sync pull`, not `confluence pull`.

## Output

Commands documented for agent parsing use `--json`. With `--json`, stdout should contain exactly one JSON value. Progress, warnings, and human hints belong on stderr so callers can parse stdout directly.

Current JSON-oriented commands include:

| Command                                 | Notes                                                          |
| --------------------------------------- | -------------------------------------------------------------- |
| `jira version list --json`              | Strips resolved email addresses unless `--emails` is supplied. |
| `jira version get --json`               | Requires a numeric version id.                                 |
| `jira version related-work list --json` | Returns related-work links for a version.                      |
| `jira version related-work add --json`  | Remote write; returns the created related-work item.           |
| `codecommit pr list --json`             | Emits JSON for single-account and filter-preset listing.       |
| `jcf issue list --json`                 | Emits issue list output for timer selection workflows.         |

## Mutability

Command references classify commands as read-only, local write, or remote write. Agents should confirm explicit intent before running remote write commands.

| Class        | Definition                                                        | Agent guidance                                                      |
| ------------ | ----------------------------------------------------------------- | ------------------------------------------------------------------- |
| Read-only    | Reads state without changing local files or remote systems.       | Safe as a first inspection step.                                    |
| Local write  | Creates or updates files/config on the machine.                   | Confirm target paths when broad writes are possible.                |
| Remote write | Mutates Jira, Confluence, Clockify, CodeCommit, or AWS resources. | Require explicit user intent and prefer `--dry-run` when available. |

## Dry Runs

`--dry-run` is only used when the command can truthfully preview the changes it would make.

Known dry-run commands:

| Command                              | Previewed operation                                  |
| ------------------------------------ | ---------------------------------------------------- |
| `confluence sync push --dry-run`     | Remote Confluence page updates that would be pushed. |
| `knpkv-skills install --dry-run`     | Skill directories that would be installed.           |
| `<product> skills install --dry-run` | Product-specific skill installation plan.            |

## Authentication

Atlassian CLIs support OAuth commands and can be inspected together with the unified `atlassian` profile manager:

```bash
jira auth create
jira auth configure
jira auth login
jira auth status

confluence auth create
confluence auth configure
confluence auth login
confluence auth status

atlassian profiles list
atlassian profiles doctor
atlassian profiles use <profile>
atlassian profiles migrate
atlassian auth refresh
```

`atlassian profiles use <profile>` only switches profiles already present in each tool auth store; it does not copy OAuth tokens across stores. Jira Clockify consumes the `jira-cli` auth store, so unified output reports that relationship.

Confluence also accepts legacy token environment variables before falling back to OAuth:

```bash
CONFLUENCE_EMAIL=you@example.com
CONFLUENCE_API_KEY=<api-token>
```

Jira Clockify composes Jira OAuth from `@knpkv/jira-cli` with a Clockify API key stored in `~/.jcf/clockify.json`.

## Paths and Local State

| Tool           | Local state                                                                                        |
| -------------- | -------------------------------------------------------------------------------------------------- |
| `jira`         | OAuth/config state from shared Atlassian helpers; issue exports default to `./jira-tickets`.       |
| `confluence`   | OAuth/config state plus `.confluence/config.json` and cloned Markdown workspace files.             |
| `codecommit`   | `~/.codecommit/config.json` and `~/.codecommit/cache.db` used by the TUI/web UI.                   |
| `jcf`          | `~/.jcf/config.json`, `~/.jcf/clockify.json`, and timer state.                                     |
| `knpkv-skills` | `${CODEX_HOME:-$HOME/.codex}/skills` and `${CLAUDE_HOME:-$HOME/.claude}/skills` unless overridden. |

Shared Atlassian helpers use `${XDG_CONFIG_HOME:-$HOME/.config}/atlassian/<tool>/auth.json` and `oauth.json` with owner-only file permissions.
