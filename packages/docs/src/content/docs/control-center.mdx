`@knpkv/control-center` is a local delivery application that joins release identity, people, source freshness, provider connections, and release-aware agents without exposing server capabilities to browser code.

## Start locally

```bash
pnpm --filter @knpkv/control-center build
pnpm --filter @knpkv/control-center start
```

Open `http://127.0.0.1:4173` and enter the single-use pairing code printed by the server. The Vite development server is intentionally loopback-only; use the authenticated application server and the documented TLS proxy setup for another machine.

## Release threads

Open a release, then choose **Ask Relay**. Each canonical release page keeps the release codename, collaborators, lifecycle, version, and source freshness visible before the first question.

The browser sends a bounded prompt and local thread history to a typed, CSRF-protected endpoint. The server derives the workspace from the authenticated session, loads the exact release projection, and invokes the configured local CLI read-only. The response includes the release projection and event cursor used for the turn, so answers cannot silently substitute another release.

Release threads currently live in the browser tab. Codex and Claude invocations are ephemeral; their internal session identifiers are not product persistence.

## Delivery graph

Control Center stores delivery traceability as a normalized graph rather than matching Jira keys, branch names, or labels in the browser. Exact source revisions become immutable entity projections. Releases, environments, and provider entities become explicit nodes, including intentionally missing nodes when a required relationship has not been resolved.

Directional relationships connect Jira work, pull requests, pipeline executions, deployments, pages, and time entries. Every relationship revision records lifecycle, confidence, provenance, actor, release or environment scope, and references to immutable evidence claims. Evidence observations have independent verification, validity, and retention metadata, so an expired observation can stop proving readiness without deleting its audit history.

### CodePipeline adapter boundary

The production AWS CodePipeline adapter is currently read-only. For one configured profile, region, and pipeline it Schema-decodes direct `distilled-aws` responses and emits stable pipeline, execution, stage, and action entities through the vendor-neutral plugin contract. Execution synchronization uses one provider execution per atomic plugin page, opaque replayable checkpoints, at most 20 pages per run, and no more than two concurrent provider calls. Action history is independently capped at 100 entries per provider page, five pages, and 200 actions per execution; truncated snapshots say so explicitly.

Artifact observations contain only names plus S3 bucket/key metadata marked for a future authenticated proxy. Resolved action configuration and credential-bearing provider, revision, artifact, or external-execution URLs are discarded. Start, stop, manual approval, retry, log content, artifact content, receipts, cancellation, and reconciliation are not advertised by this first adapter slice.

Server workflows use the durable module through two operations: a bounded atomic `write` and a tagged `read`. SQL rows, digests, table joins, and compatibility handling for older pipeline records remain private implementation details. Browser views consume later authenticated projections; they do not receive raw persistence access.

Portfolio readiness uses a bounded `effect-qb` plan from `@knpkv/control-center-sql` to read all requested release heads in one transaction. The repository still Schema-decodes every row, verifies digest-bound materialization, and quarantines malformed persisted data before it reaches the application.

The MVP database schema is deliberately unstable. Fresh databases are created from one checked-in schema snapshot; existing databases and backups must match it exactly. Schema changes may require recreating local development data. Versioned migrations begin only after the model is stable and a released database must remain readable by a newer build.

## Find workspace items

The owner-only **Items** route searches current Jira work, pull requests, Confluence pages, pipeline executions, deployments, and time entries across the authenticated workspace. Text, owner, service, type, and lifecycle filters run on the server before the 500-item response bound, so a match cannot be hidden behind an unrelated client-side prefix. Active entity-scoped human ownership assignments provide canonical names and avatar fallbacks; genuinely unassigned work remains labeled **Unassigned**. The response includes authoritative matched and workspace totals plus explicit result and owner-facet truncation flags.

Press **⌘K** on macOS or **Ctrl+K** elsewhere from any workspace page to combine the authenticated release portfolio and current item index in one compact command search. Six deterministically ranked matches remain keyboard-selectable, while **Open item results** continues on the URL-backed Items route and **Ask Relay from here** preserves the calling page or opens its exact release thread.

Filters remain in the URL for refresh and Back navigation. An item linked to one visible release opens that complete delivery context directly. Items linked to multiple releases retain every bounded current membership and ask you to choose the exact release; current unlinked items stay visible in the workspace index.

Inside the chosen release, the selected object includes a compact delivery trace centered on that exact Jira issue, pull request, page, pipeline execution, deployment, or time entry. The trace distinguishes incoming and outgoing relationships and shows the connected object, lifecycle, confidence, and immutable evidence count from the bounded current release slice.

Choose **Details** on a relationship to open its URL-addressable side panel. The panel reads the immutable relationship ledger and the exact evidence observations referenced by the selected revision. It keeps actor, provenance, lifecycle reasons, observation freshness, verification, validity, and retention visible without replacing the compact release view.

## Share one exact item

Workspace owners can open an item on the **Items** route and create an authenticated link for one explicit person. Every grant names the exact normalized entity, grantee, expiry, creating owner session, and later revocation separately in durable server state. The link contains only a random share identifier; it never carries the session cookie, CSRF proof, or another bearer credential.

Opening `/shares/<workspace-id>/<share-id>` first recovers the browser session, including cookie-authenticated reads when mutation-proof browser storage is unavailable, then rechecks the URL workspace, active grantee, expiry, revocation, and current target projection in one server-side snapshot. A valid link returns only that current item projection. The share page removes ordinary workspace navigation, and watcher sessions cannot use the workspace Items, portfolio, live-event, plugin, media, or delivery-graph read surfaces. Workspace owners and approvers retain the broader reads required by their existing roles. Release membership, relationships, evidence, and adjacent entities are not included and cannot expand the grant. A mismatched, inactive, or wrong-workspace person, expired or revoked grant, deleted target, or stale target projection shows a recoverable unavailable state without substituting another workspace or demo data.

The current Items control offers one-hour, one-day, and seven-day expiries and can revoke the newly created link immediately. If a create response is lost, retry preserves the original share identifier and expiry so the durable grant remains idempotent. Broader share administration and release-scoped grants remain future work.

## CodeCommit adapter milestone

The first production CodeCommit adapter is a read-only vertical slice for one configured AWS profile, region, and repository. It normalizes open pull requests, exact PR/base/head revisions, and every cursor-paginated changed-file entry through `entity.read@1`, `sync.incremental@1`, and `diff.inventory@1`. The owning `@knpkv/codecommit-core` package performs the real provider calls and Schema-decodes responses before Control Center normalization.

This slice does not yet register connections in the production runtime catalog or expose provider setup in Settings. Diff content, binary/generated/oversized detection, commits/history, comments, checks, review actions, approval, merge, receipts, and reconciliation remain deferred. No mutation capability is advertised, so the adapter cannot invoke CodeCommit writes.

## Read Jira issues

The server entry exports `makeJiraReadPluginRuntime`, a production `entity.read` adapter backed by the shared `@knpkv/jira-api-client`. Configure a root Jira web URL, activity page size, maximum pages, and per-request timeout; supply credentials only through the Jira client layer.

Reads normalize the issue description, environment, workflow fields, versions, parent and subtasks, comments, changelog, and human collaborators into one bounded `jira.issue` event. Collaborators are deduplicated across assignee, reporter, creator, comment author, editor, and change author roles while preserving avatar URLs. Comments and history paginate independently, stop at their configured request limit, and expose explicit truncation metadata. Oversized fixed fields are omitted deterministically and named in `truncatedFields` rather than failing the read. Cancelling the scoped read interrupts the in-flight Jira request.

This first production slice is read-only. Workspace JQL synchronization, separate `UpsertPerson` sync events, attachment content, and governed Jira mutations remain deferred rather than being represented as supported capabilities.

## Configure local providers

Local providers are disabled by default. Install and authenticate a CLI separately, then explicitly select the allowed server-side providers and an owner-controlled, least-privilege checkout:

```bash
CONTROL_CENTER_AGENT_PROVIDERS=codex,claude \
CONTROL_CENTER_AGENT_CWD=/srv/workspaces/payments \
pnpm --filter @knpkv/control-center start
```

Leave `CONTROL_CENTER_AGENT_PROVIDERS` unset to disable agent execution. Optional executable and model overrides are available through:

- `CONTROL_CENTER_AGENT_CODEX_EXECUTABLE`
- `CONTROL_CENTER_AGENT_CODEX_MODEL`
- `CONTROL_CENTER_AGENT_CLAUDE_EXECUTABLE`
- `CONTROL_CENTER_AGENT_CLAUDE_MODEL`

`CONTROL_CENTER_AGENT_CWD` is required whenever a provider is enabled; Control Center never falls back to its launch directory.

Provider paths and options remain server-only. The public API accepts only `codex | claude`, the exact release identifier, a bounded prompt, and bounded history. Only workspace owners over trusted TLS or loopback may execute the local agent; insecure-LAN viewing and watcher sessions cannot. Read-only mode prevents writes but still grants the CLI visibility into its configured checkout, so that directory must contain only material the owner intends the model to read. The server also enforces a dedicated low-rate request budget, subprocess cancellation, and bounded output.

## Review prevention proposals

When Relay is asked for a code review, each actionable finding also asks for a prevention mechanism: an existing or proposed ast-grep rule, ESLint rule, type check, focused test, or repository agent instruction. These are proposals only. Static-analysis changes still require explicit human approval and should include invalid and nearby valid fixtures.
